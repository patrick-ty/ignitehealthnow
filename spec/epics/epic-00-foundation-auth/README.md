# Epic 00 — Foundation & Authentication

## Goal
Provide a production-shaped skeleton and authentication foundation so that:
- Mobile (Flutter) and Web (Next.js) can register/login/logout via Supabase Auth
- Unified API (FastAPI) can authenticate requests using Supabase JWTs
- A required user profile exists and is isolated in `pii` schema
- The system is spec-first and agent-safe

## Non-goals (explicit)
- No journaling flows (Epic 01+)
- No community posting/messaging UI (later epic)
- No payments
- No PHI file upload pipelines (later epic)

## Canonical identity decisions
- Supabase Auth is the system of record for identity (email, phone, session/JWT).
- App database stores user profile in `pii.user_profile` keyed by `user_id` (Supabase UUID).
- Health/journal data MUST NOT be stored in `pii.*` tables.

## Required profile fields (blocking gate)
Users must complete profile before entering the main app experience:
- first_name (required)
- last_name (required)
- email (required, via Supabase Auth)
- mobile (required)
- zipcode (required)
- birth_year (required, reasonable bounds)

Note: birth_year is stored in plaintext and is not encrypted, as it is not directly identifying on its own and is required for longitudinal health analysis. Birth month and exact date of birth are intentionally not collected.

Community identity fields:
- display_name (optional; defaults to handle; NOT required to be unique)
- handle (system-generated; REQUIRED; UNIQUE; may be surfaced later in community UX)

## Handle & Display Name Rules (Canonical)

### Handle
- **Required**
- **Auto-generated by the API at profile completion**
- **Plaintext (not encrypted)**
- **Globally unique**
- **Stable once created**
- **Users cannot edit or regenerate their handle in v1**

#### Generation rules
- Handle is generated as:  
  **`<allowlisted_word><5-digit number>`**  
  Examples:
  - `juniper48291`
  - `willow90344`
  - `luna11720`
- The word component:
  - Comes from a **curated allowlist** of neutral, non-PII English words
  - Is intentionally **feminine-adjacent but inclusive**
  - Avoids:
    - names
    - body terms
    - medical terms
    - emotional diagnoses
    - religious/spiritual terms
    - slang or profanity
- The numeric suffix:
  - Is randomly generated
  - Exists solely to ensure uniqueness
- A **denylist and reserved-word list** is enforced to block:
  - platform terms (`admin`, `support`, `staff`, `ignite`, etc.)
  - medical roles (`doctor`, `nurse`, `patient`)
  - offensive or inappropriate words
- Collisions are handled via retry with a capped number of attempts; failure is treated as a system error.

#### Governance
- Allowlist, denylist, and reserved words live in **repository-managed files**, not hardcoded logic.
- Wordlists are auditable and safely editable without changing core logic.

---

### Display Name
- **Optional**
- **Plaintext (not encrypted)**
- **Defaults to the generated handle**
- **Never defaults to first_name, last_name, or any decrypted PII**
- User may change display_name at any time
- display_name:
  - Does **not** need to be unique
  - May contain user-entered text (including names), at the user’s discretion

> Note: If a user chooses to enter personal information into display_name, they are doing so explicitly. The system does not infer or inject PII into display_name automatically.

---

### Privacy & Intent
- Handles are designed to be **non-identifying by default**
- No handle or display_name is derived from encrypted profile fields
- This separation protects user privacy while enabling future community features

---

### Explicitly Removed (do not implement)
- ❌ Name-derived handles (e.g., `patrick027`)
- ❌ Automatic use of first_name or last_name for display_name
- ❌ Encrypted handles or display names
- ❌ User-editable handles in v1

## Profile extensions (future-proofing)
Additional profile information may be stored via key/value properties:
- keys are allowlisted and namespaced (prefs.*, onboarding.*, notifications.*, community.*)
- values stored as JSON (jsonb)
- PHI/health data MUST NOT be stored in profile properties

## Security baseline
- API must validate Supabase JWT (signature, issuer, expiry; audience if configured).
- Default to least privilege; no sensitive payload logging.
- All profile read/write must be tenant-isolated (RLS or API-layer checks).
